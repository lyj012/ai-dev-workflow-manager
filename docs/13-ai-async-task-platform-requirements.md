# AI 异步任务平台需求文档

## 1. 背景

根据《项目方向复盘》，旧方向“个人 AI 开发 workflow 手填管理器”已经停止，不再继续投入阶段手填、workflow 模板配置、手动推进流程和旧页面美化。

项目后续重构为：

```text
AI 异步任务分析与执行平台
```

本需求文档作为后续代码开发依据。后续开发必须优先遵循本文档，而不是继续沿用旧 workflow 管理器的产品假设。

## 2. 产品定位

新方向不是“流程记录工具”，而是一个后端工程能力驱动的 AI 任务平台。

第一阶段重点不是接入真实模型，而是先跑通：

```text
任务提交
↓
异步执行
↓
状态流转
↓
日志追踪
↓
结果保存
↓
失败重试
```

第一版不追求智能效果，先把异步任务平台的工程闭环做稳定。

## 3. 第一阶段目标

第一阶段目标是完成一个 mock AI 异步任务 MVP：

```text
用户输入需求
↓
前端创建任务
↓
后端保存任务，状态 CREATED
↓
后端异步线程开始执行
↓
状态变为 ANALYZING
↓
模拟等待 3 秒
↓
生成模拟分析结果
↓
状态变为 SUCCESS
↓
前端展示状态、日志和结果
```

如果执行失败：

```text
状态变为 FAILED
↓
记录失败日志和错误信息
↓
用户点击重试
↓
重新进入异步执行流程
```

## 4. 开发顺序

本次重构开发顺序固定为：

```text
先搭前端架子
↓
再开发后端功能
↓
最后前后端联调
```

原因：

- 先通过前端确定新产品的主操作路径。
- 避免后端先建复杂表和接口后，前端发现流程不成立。
- 第一阶段后端是 mock AI，前端先用 mock 数据即可验证页面结构和用户路径。

## 5. P0 范围

P0 只做新方向最小闭环，不再扩展旧 workflow。

### 5.1 前端 P0

新增 AI 任务模块页面，先使用前端 mock 数据或临时本地状态，不依赖后端完成。

必须包含：

- AI 任务列表页。
- AI 任务创建页或创建面板。
- AI 任务详情页。
- 任务状态展示。
- 执行日志展示。
- 分析结果展示。
- 失败任务重试按钮。
- 任务状态刷新入口。

前端第一阶段只验证页面路径，不做复杂交互。

### 5.2 后端 P0

新增 AI 异步任务模块，先使用 mock AI 执行器。

必须包含：

- 创建 AI 任务接口。
- 查询 AI 任务列表接口。
- 查询 AI 任务详情接口。
- 查询 AI 任务日志接口。
- 查询 AI 任务结果接口。
- 重试失败任务接口。
- 异步执行服务。
- 状态流转控制。
- 执行日志记录。
- 模拟分析结果保存。

## 6. 暂不做内容

第一阶段明确不做：

- 真实 AI API 接入。
- 多模型路由。
- RAG。
- 代码扫描。
- Codex CLI / Claude CLI 接入。
- 权限系统。
- 模板市场。
- 团队协作。
- 复杂统计大屏。
- 旧 workflow 模板配置增强。
- 旧阶段流程区继续优化。

如果开发过程中发现旧 workflow 相关页面需要适配，只做入口降级或隐藏，不继续增强旧功能。

## 7. 前端需求

### 7.1 页面结构

新增前端模块建议命名为：

```text
AI 任务
```

推荐路由：

```text
/ai-tasks
/ai-tasks/:taskId
```

可以先不做独立创建页，在列表页顶部放创建面板或弹窗。

### 7.2 AI 任务列表页

页面目标：

```text
让用户看到所有 AI 异步任务的执行状态，并能创建新任务。
```

列表字段：

- 任务 ID。
- 任务标题。
- 任务状态。
- 重试次数。
- 创建时间。
- 更新时间。
- 操作。

操作：

- 查看详情。
- 失败任务可重试。
- 刷新列表。

状态展示建议：

- `CREATED`：已创建。
- `ANALYZING`：分析中。
- `SUCCESS`：执行成功。
- `FAILED`：执行失败。

### 7.3 AI 任务创建

创建表单字段：

- 任务标题，必填。
- 需求描述，必填。
- 任务类型，可选，第一阶段可用普通下拉。
- 备注，可选。

提交后预期：

- 前端调用创建接口。
- 创建成功后进入任务详情页，或刷新列表并展示新任务。
- 如果后端尚未完成，前端先使用 mock 数据模拟创建成功。

### 7.4 AI 任务详情页

页面目标：

```text
展示一个异步任务从创建到完成的完整执行过程。
```

详情页分区：

- 任务基础信息。
- 当前状态。
- 执行日志。
- 分析结果。
- 失败信息。
- 操作区。

操作区：

- 刷新任务。
- 失败时重试。
- 返回列表。

如果任务状态为 `ANALYZING`，前端可以提供手动刷新按钮。第一阶段不要求 WebSocket 或 SSE。

### 7.5 执行日志展示

日志字段：

- 时间。
- 日志级别。
- 执行节点。
- 日志内容。

示例：

```text
17:30:01 INFO TASK_CREATED 任务已创建
17:30:02 INFO TASK_ANALYZING 开始模拟 AI 分析
17:30:05 INFO TASK_SUCCESS 模拟分析完成
```

前端展示要求：

- 按时间升序展示。
- 日志内容可换行。
- 错误日志突出显示，但不要过度装饰。

### 7.6 分析结果展示

第一阶段 mock 分析结果可以包括：

- 需求摘要。
- 风险点。
- 建议执行步骤。
- 建议验证方式。

展示要求：

- 没有结果时显示空状态。
- `SUCCESS` 后展示结果。
- `FAILED` 时展示失败信息，不展示伪结果。

## 8. 后端需求

### 8.1 模块命名

新增后端模块建议命名为：

```text
aitask
```

推荐包结构：

```text
com.aidev.workflowmanager.aitask.controller
com.aidev.workflowmanager.aitask.service
com.aidev.workflowmanager.aitask.domain
com.aidev.workflowmanager.aitask.mapper
com.aidev.workflowmanager.aitask.dto
com.aidev.workflowmanager.aitask.vo
```

### 8.2 状态枚举

第一阶段只保留 4 个核心状态：

```text
CREATED
ANALYZING
SUCCESS
FAILED
```

状态含义：

- `CREATED`：任务已创建，尚未开始执行。
- `ANALYZING`：异步任务执行中。
- `SUCCESS`：任务执行成功，并已保存结果。
- `FAILED`：任务执行失败，已记录错误信息。

### 8.3 状态流转

允许的状态流转：

```text
CREATED -> ANALYZING
ANALYZING -> SUCCESS
ANALYZING -> FAILED
FAILED -> CREATED
FAILED -> ANALYZING
```

说明：

- 创建任务后可以立即提交异步执行。
- 重试失败任务时，可以先重置为 `CREATED`，再进入 `ANALYZING`。
- `SUCCESS` 状态第一阶段不允许重试。
- `ANALYZING` 状态不允许重复触发执行。

### 8.4 异步执行规则

第一阶段使用 mock AI 执行器，不调用真实模型。

执行流程：

```text
记录开始日志
↓
更新状态 ANALYZING
↓
sleep 3 秒
↓
生成模拟结果
↓
保存结果
↓
更新状态 SUCCESS
↓
记录完成日志
```

失败模拟：

- 第一阶段可以先不做随机失败。
- 后续可以增加测试开关或特定输入触发失败。

### 8.5 幂等和并发约束

必须避免：

- 同一个任务同时执行多次。
- `ANALYZING` 状态重复点击重试。
- `SUCCESS` 任务被错误重试。
- 失败重试时旧结果和新结果混乱。

第一阶段最小处理：

- 重试接口只允许 `FAILED` 状态。
- 执行前检查当前状态。
- 保存日志时记录任务 ID 和状态节点。
- 重试时可清理旧错误信息，但保留历史日志。

## 10. API 草案

接口统一挂在：

```text
/api/ai-tasks
```

### 10.1 创建任务

```text
POST /api/ai-tasks
```

请求字段：

- `title`
- `requirement`
- `taskType`
- `remark`

响应：

- 任务详情。

### 10.2 查询任务列表

```text
GET /api/ai-tasks
```

查询参数：

- `status`
- `pageNo`
- `pageSize`

响应：

- 分页任务列表。

### 10.3 查询任务详情

```text
GET /api/ai-tasks/{taskId}
```

响应：

- 任务基础信息。
- 当前状态。
- 错误信息。
- 重试次数。
- 最新结果摘要。

### 10.4 查询任务日志

```text
GET /api/ai-tasks/{taskId}/logs
```

响应：

- 日志列表。

### 10.5 查询任务结果

```text
GET /api/ai-tasks/{taskId}/result
```

响应：

- 分析结果。

### 10.6 重试失败任务

```text
POST /api/ai-tasks/{taskId}/retry
```

约束：

- 仅 `FAILED` 状态允许重试。
- 重试后进入新的异步执行流程。

响应：

- 最新任务详情。

## 11. 前后端联调路径

前端先使用 mock 数据完成页面架子。

后端接口完成后，按以下顺序替换：

```text
任务列表 mock -> GET /api/ai-tasks
任务创建 mock -> POST /api/ai-tasks
任务详情 mock -> GET /api/ai-tasks/{taskId}
日志 mock -> GET /api/ai-tasks/{taskId}/logs
结果 mock -> GET /api/ai-tasks/{taskId}/result
重试 mock -> POST /api/ai-tasks/{taskId}/retry
```

联调必须验证：

- 创建任务后能看到 `CREATED` 或 `ANALYZING` 状态。
- 等待异步执行后能看到 `SUCCESS` 状态。
- 日志按时间展示。
- 成功后展示结果。
- 失败后展示错误信息。
- 失败任务可以重试。
- 非失败任务不能重试。

## 12. 验收标准

### 12.1 前端架子验收

前端架子完成时必须满足：

- 能进入 AI 任务列表页。
- 能创建 mock AI 任务。
- 能进入任务详情页。
- 能展示不同状态。
- 能展示日志和结果区域。
- 能展示失败重试入口。
- `npm run build` 通过。

### 12.2 后端功能验收

后端功能完成时必须满足：

- 创建任务接口可用。
- 任务创建后能触发异步执行。
- 状态能从 `CREATED` 流转到 `ANALYZING`，再到 `SUCCESS`。
- 执行日志被保存。
- 模拟分析结果被保存。
- 失败状态可记录。
- 失败任务可重试。
- `mvn test` 通过。

### 12.3 联调验收

前后端联调完成时必须满足：

- 前端创建任务后，后端真实保存任务。
- 前端能看到异步状态变化。
- 前端能刷新看到日志和结果。
- 前端重试失败任务时，后端真实重新执行。
- 前端不再依赖 mock 数据完成主链路。

## 13. 风险点

### 13.1 异步状态不一致

风险：

```text
前端显示执行中，但后端任务已经失败或成功。
```

控制方式：

- 前端提供刷新入口。
- 后端状态以数据库为准。
- 日志记录每次状态变化。

### 13.2 重复执行

风险：

```text
用户重复点击导致同一个任务并发执行。
```

控制方式：

- 后端执行前校验状态。
- `ANALYZING` 不允许重试。
- `SUCCESS` 不允许重试。

### 13.3 结果覆盖

风险：

```text
失败重试后，新旧结果混乱。
```

控制方式：

- 第一阶段失败任务不生成结果。
- 重试成功后更新或重建结果。
- 历史执行过程通过日志追踪。

### 13.4 旧功能干扰

风险：

```text
旧 workflow 页面继续占据项目主线，导致新方向不清晰。
```

控制方式：

- 新开发只围绕 `AI 任务` 模块。
- 旧模块暂时保留，但不继续增强。
- 后续 README 和演示文档改为新方向。

```text

↓

↓
最后完成真实前后端联调
```

旧 workflow 管理器不再作为产品主线。

本需求文档是后续开发的主要依据。
